一、微服务

  1. 定义：微服务架构是一种应用程序开发风格，一个单一的应用程序被分解为多个小服务，这些小服务在各自独立的进程中运行，且通过轻量级通信机制（HTTP RESTFUL API）进行通讯。
  2. 特征：
    • 易于维护：微服务规模小，应该只处理单个业务任务。
    • 独立的扩展和部署：微服务有其各自的部署模式和节奏。因此，每个服务都可以根据该服务应该满足的负载进行扩展。每个服务都可以根据其计划进行部署。
      • 每个服务都可以独立的部署到各种环境中。（开发、测试、生产环境）
      • 每个服务都能独立启动或销毁而不会对其他服务造成影响。
      • 当用户量和并发量的增加时，我们还可以将微服务集群化部署，从而增加系统的负载能力。
    • 独立技术的使用：微服务的代码库与部署环境分离，因此可以根据用例决定微服务需要使用的语言和技术。无需在所有微服务中使用通用堆栈。
      • 即不同的服务可以采用不同的技术、编程语言。

  • 兼容性：

    • Spring Boot 3—Spring Security 6.2.x
    • Spring Boot 2—Spring Security 5.8.x

  • Spring Security 是一个提供身份认证、授权和防护常见攻击的框架。

    • 身份认证（Authentication）：

      • 验证用户身份（如用户名/密码、短信验证等），解决“你是谁”的问题‌；
      • 通过UserDetailsService接口实现自定义用户数据源查询‌；
      • 支持表单登录、OAuth2、LDAP等多种认证方式，默认提供基础表单登录页‌；
      • 密码加密存储需使用PasswordEncoder接口实现类（如BCrypt算法）‌；

    • 权限控制（Authorization）

      • 根据用户权限控制资源访问（如角色分配、URL权限控制），解决“你能做什么”的问题‌
      • 基于角色（Role）或权限（Permission）的访问控制‌；
      • URL级别‌：通过配置HttpSecurity限制访问路径‌；
      • 方法级别‌：使用@PreAuthorize、@Secured等注解实现细粒度控制‌
      • 常用注解：@PreAuthorize、@Secured、@PostAuthorize‘

    • 安全防护机制

      • ‌CSRF防护：默认启用，需配合模板引擎或手动配置令牌‌；

      • 会话管理‌：防止会话固定攻击，支持并发会话控制‌；

      • HTTP安全头：自动配置X-Content-Type-Options等头部增强防护‌；

一、MyBatis

  1. MyBatis是一个开源、轻量级的数据持久化框架，内部封装了JDBC。

  2. MyBatis核心对象：

    （1）核心接口和类：

    • SqlSessionFactoryBuilder：临时工具类，用完即销毁，无状态，无需长期持有。

    • SqlSessionFactory：线程安全，与应用同生命周期，通常只初始化一次，避免重复解析配置；缓存数据库配置、Mapper 映射等元信息；由 IoC 容器管理其生命周期。

    • SqlSession：线程不安全，需在方法内创建并使用后立即关闭；代表一次数据库会话，用于执行 SQL、管理事务、获取 Mapper 接口。

      • 构建阶段‌：SqlSessionFactoryBuilder 读取配置文件 → 创建 SqlSessionFactory。

      • 运行阶段‌：SqlSessionFactory 生产 SqlSession → SqlSession 执行 SQL。

      • 销毁阶段‌：SqlSession 执行完毕后关闭，SqlSessionFactory 随应用关闭销毁。

一、Spring Data JPA 与 Hibernate 的核心区别

  1. 区别

    • ‌Hibernate‌：

      • 是独立的 ‌ORM 框架‌，直接实现 JPA 规范，负责将 Java 对象映射到数据库表，并管理 SQL 生成、缓存、事务等底层操作‌。
      • 提供 HQL（面向对象的查询语言）和 Criteria API （动态条件构建）等高级查询工具‌。
      • 需手动编写查询逻辑，灵活性高但开发效率较低‌。
      • 需通过 Transaction 对象显式控制事务边界（如 begin()、commit()）‌。

    • ‌Spring Data JPA‌：

      • 是 ‌Spring 生态的子模块‌，基于 JPA 规范构建的 ‌数据访问层抽象‌，旨在简化 DAO 层开发‌。
      • 依赖 Hibernate 作为默认的 JPA 实现，但可与其他 JPA 兼容框架（如 EclipseLink）集成‌。
      • 通过 ‌方法命名规则‌（如 findByUsernameAndEmail）自动生成查询‌。
      • 支持 @Query 注解自定义 JPQL 或原生 SQL，平衡灵活性与简洁性‌。
      • 与 Spring 事务管理深度集成，支持声明式事务（@Transactional 注解）。

一、Spring Boot：

  1. Spring Boot是什么？

    • 微框架—微服务。

    • Spring Boot = Spring + Tomcat/Jetty(Http Servers) + XML(Configuration)

    • 特征：

      • Spring Boot集成了大量常用的第三方库配置，零配置开箱即用；

      • Spring Boot不需要很多XML配置；

      • Spring Boot旨在尽可能地快速启动和运行，需要最少的Spring前期配置；

        Spring Boot：Build Anything， Build任何东西；

        SpringCloud：Coordinate Anything，协调任何事情；

一、前后端开发历程：

  1. 三层架构：

    ‌发展原因‌：

    • ‌职责解耦‌：表现层、业务层、数据层，通过分层实现职责分离，降低代码耦合度，提升可扩展性‌。
    • ‌协作效率‌：开发团队可分工协作，例如后端专注于业务逻辑与数据，前端负责界面交互，减少开发冲突‌。
    • ‌复用性‌：业务逻辑层独立于表现层，可复用至不同终端（如 Web、移动端）‌

    ‌局限性‌：

    • ‌Servlet 耦合‌：传统三层架构中，Servlet 需处理多个请求逻辑，导致代码臃肿和复用性差‌；

    • ‌视图与业务未彻底分离‌：如 JSP 中仍嵌入 Java 代码，难以独立维护‌。
