一、JavaBean（Bean/豆子）

  1. 定义

    一个固定模式的Java类，用于封装数据，包含：

    ①一个无参构造函数；

    ②需要序列化（数据结构或对象转换为字节流）并实现Serializable接口；

    ③可读写属性；

    ④getter和setter方法方便外部程序读取和修改。

二、Spring是什么？能做什么？

  1. Spring是什么？（万能工具箱）

    解释①：用来装JavaBean的一种容器，即用来管理JavaBean实例化对象的一种容器。

    可以通过@bean自动完成对象的实例化。

    解释②：为了解决对象的创建和管理。

    解释③：Spring 让 Java 开发从“手工作坊”变成“现代化工厂”，专注业务逻辑，而不是重复造轮子。

  2. Spring能做什么？

    ①预定模板||依赖注入（如数据库操作只需要直接让Spring准备好即可，不需要去自己去写。）

    ②解耦、简化开发、易于测试；

    ③对事务进行声明；（事务是逻辑上的一组操作，要么都执行，要么都不执行）（如对数据库操作）等。

    如果把写代码比作造房子‌，传统方式需要自己搬砖、砌墙、接电线，而 Spring 提供了：

    • ‌预制板‌（依赖注入）：直接组装现成的模块。
    • ‌智能管家‌（AOP）：自动处理水电安装、刷墙等重复工作。
    • ‌装修模板‌（Spring Boot）：一键生成基础户型，省去画图纸的时间。
    • 用 ‌Spring Cloud‌ 可以快速构建分布式系统（比如电商中的订单、支付等独立服务），解决服务调用、负载均衡等问题。
    • 通过 ‌Spring MVC‌ 框架，可以轻松处理网页的请求、响应、路由等，让前后端协作更清晰。Spring Boot‌ 进一步简化配置，可以更快启动一个Web项目。
    • Spring 整合了 JDBC、JPA 等工具，让数据库操作（增删改查）变得更简单，减少繁琐代码。提供 ‌事务管理‌，保证数据库操作的一致性（比如转账失败时自动回滚，避免数据错误）。

三、Spring详解：

  1. 定义：

    Spring将各种功能封装为Bean/豆子，使用时自动去调用这个Bean即可

    更准确的说Spring 的核心机制确实是通过 ‌**“Bean”‌ 和 ‌“IoC（控制反转）容器”**‌ 来管理对象。

    **（1）Bean 是什么？**‌

    • ‌Bean 是 Spring 管理的对象‌：比如你的数据库工具类、网络请求处理器、配置参数等，都可以声明为 Bean。
    • ‌Bean 不是普通的 new 出来的对象‌：它由 Spring 的 IoC 容器创建、组装和管理，生命周期由容器控制。

    **（2）IoC 容器：Spring 的“智能仓库”生产零件**‌

    • ‌传统方式‌：你需要自己手动 new 对象，并层层传递依赖（比如 A 依赖 B，B 依赖 C，需要手动把 C 传给 B，再把 B 传给 A）。
    • ‌Spring 的做法‌：你只需告诉 Spring 哪些类是 Bean（通过注解或配置），‌容器会自动创建这些 Bean，并按需注入它们之间的依赖‌，就像“智能仓库”自动给你配送零件。

    **（3）依赖注入（DI）：自动“装配”Bean|拼装零件为产品**‌

    • ‌你只需要定义 Bean‌（比如用 @Component 标记一个类），然后告诉 Spring：“我的类需要某个 Bean”（比如用 @Autowired 注解）。

    • ‌Spring 会自动完成两件事：

      ①创建 Bean‌：根据配置或扫描路径，实例化对象。

      ②注入依赖‌：如果 Bean A 依赖 Bean B，Spring 会自动把 B 注入到 A 中（无需手动 A.setB(new B())）。

  5. IoC控制反转和DI依赖注入的区别：

    分工：

    ‌步骤‌ ‌执行者‌ ‌行为‌ ‌技术体现‌
    ‌Bean 的创建‌ IoC 容器 根据配置（XML/Java配置/注解）实例化 Bean ApplicationContext、BeanFactory
    ‌依赖的注入‌ DI 机制 将其他 Bean 或值注入到目标 Bean 的属性中 @Autowired、构造器参数、Setter 方法

    形象类比：

    • ‌IoC 容器‌ 像‌工厂车间‌：负责生产零件（Bean）。
    • ‌DI 机制‌ 像‌组装流水线‌：将零件（依赖的 Bean）拼装成完整产品。

    **IoC 和 DI 是协作关系：**‌

    • ‌IoC 是前提‌：没有 IoC 容器创建 Bean，DI 将无对象可注入。
    • ‌DI 是手段‌：通过依赖注入，IoC 容器才能管理对象间的复杂依赖关系。

    **总结：**‌

    • ‌IoC 创建 Bean‌：容器掌控对象的生死（实例化、销毁）。
    • ‌DI 注入 Bean‌：容器解决对象的依赖（自动装配）。

  7. AOP面向切面编程：Aspect Oriented Programming

    • **定义：**模块化功能，把程序重复的代码抽取出来，在需要执行的时候，使用代理的技术，在不修改原来代码的基础上，对已有方法进行增强。

    • 核心思想：正是通过“横向切入”的方式，在程序执行的前后（或特定位置）统一添加公共功能‌。

    • 注解：

      注解 描述
      @Aspect 把当前类声明成切面类
      @Before 把当前方法看成是前置通知
      @AfterReturning 把当前方法看成是后置通知。
      @AfterThrowing 把当前方法看成是异常通知
      @After 把当前方法看成是最终通知
      @Around 把当前方法看成是环绕通知
      @Pointcut 指定切入点表达式

      代码详解：https://www.cainiaojc.com/spring/spring-aop-aspectj-annotation-example.html

    • 核心概念：

      ‌术语‌ ‌作用‌ ‌类比‌
      ‌**切面（Aspect）**‌ 封装公共功能的模块（如日志记录工具类）。 类似一个“插件包”。
      ‌**连接点（Join Point）**‌ 程序执行过程中的特定点（如方法调用、异常抛出）。 代码中可插入切面的“锚点”。
      ‌**切点（Pointcut）**‌ 通过表达式匹配需要增强的连接点（例如所有 UserService 的方法）。 定义“在哪些锚点插入功能”。
      ‌**通知（Advice）**‌ 切面在连接点执行的具体动作（如前置日志、后置事务提交）。 插件包中的具体“功能代码”。
      目标（Target） 代理的目标对象。通俗点讲：你需要增强的类，这个类就是目标对象
      代理（Proxy） 一个类被AOP增强后，就产生一个结果代理类。

四、Spring高级：

  1. 数据库事务：

    • **原子性Atomicity：**事务是一个不可分割的工作单位，事务中的操作要么都成功，要么都失败。‌
    • **一致性Consistency：**事务前后数据的完整性必须保持一致。
    • **隔离性Isolation：**多个用户并发访问数据库时，一个用户的事务不能被其它用户的事务干扰，多个并发事务之间数据要相互隔离。
    • **持久性Durability：**一个事务一旦被提交，它对数据库中数据的改变就是永久性的，接下来即使数据库发生故障也不应该对其有任何影响。

  2. 并发问题：

    • **脏读：**事务A读取了事务B ‌未提交的修改‌，若事务B回滚，事务A读取的数据无效‌。‌
    • **不可重复读：**事务A多次读取同一数据，因事务B的 更新/删除操作已提交‌，导致两次结果不一致‌
    • **幻读：**事务A在范围查询中，因事务B ‌插入新数据并提交‌，导致两次结果集不一致‌
问题类型‌ ‌核心差异‌ ‌隔离级别要求‌ ‌技术手段‌
‌脏读‌ 读取未提交数据 READ_COMMITTED 及以上 悲观锁、提交前禁止读取‌
‌不可重复读‌ 同一数据被修改/删除 REPEATABLE_READ 及以上 MVCC、行级锁‌
‌幻读‌ 范围查询中出现新数据 SERIALIZABLE 间隙锁、索引范围锁‌

  1. 声明式事务管理：

    @Transactional 是 Spring 框架中用于 ‌声明式事务管理‌ 的核心注解，通过自动管理事务的开启、提交和回滚，简化数据库操作的原子性和一致性保障。

    （1）核心功能

    • ‌事务原子性保障‌
      确保方法内的数据库操作作为一个整体执行：‌全部成功则提交，任意失败则回滚‌，符合 ACID 原则的原子性要求‌。
    • ‌声明式事务管理‌
      替代传统编程式事务（手动提交/回滚），通过注解配置实现事务逻辑与业务逻辑的解耦‌。

    （2）关键特性

    • 传播行为（Propagation）

      定义事务的边界策略，例如：

      • REQUIRED（默认）：若当前存在事务则加入，否则新建事务‌。
      • REQUIRES_NEW：始终新建独立事务，挂起当前事务（适用需隔离的场景）‌。

    • 隔离级别（Isolation）

    控制事务并发时的数据可见性，如：

    • READ_COMMITTED：防止脏读（默认级别）‌。

    • SERIALIZABLE：最高隔离级别，避免幻读但性能较低‌。

    • 回滚规则（Rollback）

    • 默认规则‌：仅回滚 RuntimeException 和 Error‌。

    • 自定义规则‌：通过 rollbackFor 指定触发回滚的异常类型（如 rollbackFor = Exception.class）‌。‌

    • 超时与只读

    • timeout：设置事务超时时间（秒），超时自动回滚‌。

    • readOnly = true：优化查询操作，禁止写入（部分数据库支持性能优化）‌。

    （3）应用场景

    • 数据更新操作‌
      如转账业务（扣款与入账需原子性）‌。
    • ‌批量操作‌
      批量插入/更新时，事务确保整体成功或回滚‌。
    • ‌多数据源协调‌
      跨多个数据库或微服务调用时，通过分布式事务管理器扩展（需结合其他组件）‌。

    （4）使用规范

    Spring 通过 AOP 动态代理实现事务管理，‌自调用（同类方法调用）不触发事务。
