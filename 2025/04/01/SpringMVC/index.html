一、前后端开发历程：

  1. 三层架构：

    ‌发展原因‌：

    • ‌职责解耦‌：表现层、业务层、数据层，通过分层实现职责分离，降低代码耦合度，提升可扩展性‌。
    • ‌协作效率‌：开发团队可分工协作，例如后端专注于业务逻辑与数据，前端负责界面交互，减少开发冲突‌。
    • ‌复用性‌：业务逻辑层独立于表现层，可复用至不同终端（如 Web、移动端）‌

    ‌局限性‌：

    • ‌Servlet 耦合‌：传统三层架构中，Servlet 需处理多个请求逻辑，导致代码臃肿和复用性差‌；

    • ‌视图与业务未彻底分离‌：如 JSP 中仍嵌入 Java 代码，难以独立维护‌。

  2. MVC模式：

    ‌发展原因‌：

    • ‌动态交互需求‌：随着 Web 应用复杂度提升，需将用户交互（View）、业务逻辑（Model）与流程控制（Controller）分离，MVC 模式通过组件化分工提升开发效率‌。
    • ‌框架支持‌：SpringMVC框架标准化了 MVC 实现，简化请求路由、参数绑定等操作，降低开发门槛‌。
    • ‌视图灵活性‌：支持多种视图技术（如 JSP、Thymeleaf），便于适配不同展示需求‌。

    ‌局限性‌：

    • ‌前后端耦合‌：传统 MVC 中后端仍需处理视图渲染（如 JSP），导致前端依赖后端环境，难以独立开发‌。
    • ‌性能瓶颈‌：动态页面渲染加重服务器负载，影响高并发场景性能‌。

  3. 前后端分离：

    ‌发展原因‌：

    • ‌技术专业化‌：前端技术如Vue复杂度提升，需独立工程化开发，后端专注于 API 与业务逻辑，提升协作效率‌。

    • ‌解耦与性能优化

      • 静态资源（HTML/CSS/JS）由 CDN 或独立服务器托管，减少后端压力‌。
      • 异步请求（AJAX）与单页应用（SPA）提升用户体验，减少整页刷新‌。

    • ‌开发流程革新

      • 前端可本地模拟 API 响应（如 Mock.js+Node.js），不依赖后端环境，加速迭代‌。Mock.js+Node.js

      • 后端通过 RESTful API 标准化数据交互，支持多终端复用‌。

    ‌推动因素‌：

    • ‌JSP 时代痛点‌：JSP 混合 Java 与 HTML，维护困难且前后端职责不清‌。
    • ‌模块化与微服务趋势‌：前后端分离契合微服务架构，支持独立部署与扩展‌。

二、MVC是什么？

Model-View-Controller，一种软件设计模式。

  1. 模型（Model）：‌
    • ‌职责‌：管理数据和业务逻辑。
    • ‌功能
      • 存储、处理、验证数据（如数据库操作、计算逻辑）。
      • 独立于界面，不关心数据如何展示。
  2. 视图（View）：‌
    • ‌职责‌：负责用户界面的展示。
    • ‌功能
      • 将 Model 中的数据可视化（如网页、App界面）。
      • 不处理业务逻辑，仅接收用户输入并显示结果。
  3. 控制器（Controller）：‌
    • ‌职责‌：协调 Model 和 View，处理用户请求。
    • ‌功能
      • 接收用户输入（如点击按钮、提交表单）。
      • 调用 Model 处理数据，根据结果更新 View。
  4. MVC的优势：
    • ‌解耦‌：各组件职责明确，修改界面不影响业务逻辑。
    • ‌复用性‌：同一 Model 可被多个 View 使用（如网页版和移动端共享数据逻辑）。
    • ‌协作友好‌：开发者可分工开发不同组件（如前端专注 View，后端专注 Model）。

  6. 注解：

    • @Controller‌：标注一个类为控制器，用于处理 HTTP 请求‌；
    • ‌**@RestController**‌：相当于@Controller + @ResponseBody，用于返回JSON或者XML数据；
    • @RestquestMapping(“/路径”)：映射请求路径，用于类或者方法；
    • **@GetMapping **：等价于@RequestMapping(method = RequestMethod.GET)；
    • **@PostMapping **：等价于@RequestMapping(method = RequestMethod.POST)；
    • @PutMapping：整个资源更新，等价于@RequestMapping(method = RequestMethod.PUT)；
    • @DeleteMapping：等价于@RequestMapping(method = RequestMethod.DELETE)；
    • @PatchMapping：部分资源更新，等价于@RequestMapping(method = RequestMethod.PATCH)；
    • @RequestParam：获取请求参数（常见于查询参数）；
    • @PathVariable：获取路径中的动态参数；
    • @RequestBody：解析请求体中的 JSON 或 XML 数据，并将其绑定到对象；
    • @RequestHeader：获取 HTTP 请求头的值；
    • @CookieValue：获取 Cookie 值；
    • @ResponseBody：将返回结果直接写入 HTTP 响应体（通常用于返回 JSON 或 XML 数据）；
    • @ModelAttribute：将方法返回值或请求参数绑定到模型中，在视图渲染时可用；
    • @Valid：对方法参数进行校验（需要配合 Java Bean Validation API）；
    • @SessionAttributes：将某些模型属性存储到会话中，便于后续访问；
    • @RequestPart：处理 multipart/form-data 请求，常用于文件上传；
    • @ExceptionHandler(异常类型.class) ：指定要捕获的异常类型；
    • @ControllerAdvice：一个全局异常处理器，适用于所有控制器；
    • @RestControllerAdvice:@ControllerAdvice 的变体，返回值默认是 JSON 格式；
    • @ResponseStatus:异常类上添加该注解，指定 HTTP 状态码和原因。

  9. 拦截器Interceptor：

    （1）拦截器是一种动态拦截请求并对其进行预处理和后处理的机制，类似于 Servlet 的过滤器（Filter）；拦截器的核心是实现 HandlerInterceptor 接口或继承其实现类 HandlerInterceptorAdapter，并通过配置注册到应用程序中。

    （2）用途：

    • 日志记录：记录请求的处理时间或访问记录。
    • 权限校验：检查用户是否有权限访问某些资源。
    • 参数预处理：对请求参数进行统一校验或修改。
    • 响应后处理：在响应数据返回之前进行修改或处理。
    • 全局异常处理：捕获和处理控制器未捕获的异常。

    （3）HandlerInterceptor 接口定义了拦截器的三个核心方法：

    • preHandle：权限校验、日志记录、参数验证或处理、短路机制（按注册顺序执行。）
    • postHandle：修改返回数据、日志记录、动态调整视图、扩展功能（按注册顺序的逆序执行。）
    • afterCompletion：资源清理、日志记录、性能监控、异常处理或补偿（按注册顺序的逆序执行。）

    （4）拦截器的实现：

    实现 HandlerInterceptor 接口或继承其默认实现类 HandlerInterceptorAdapter。

    （5）拦截器的注册：

    通过实现 WebMvcConfigurer 来完成。

    （6）拦截器和过滤器的区别：

    特性 拦截器（Interceptor） 过滤器（Filter）
    工作层级 SpringMVC 层 Servlet 容器层
    拦截对象 控制器（Handler） 所有请求（静态资源也可拦截）
    功能扩展 可以访问 Spring 上下文，支持依赖注入 无法直接访问 Spring 上下文
    配置方式 Spring 配置类 web.xml或 @WebFilter注解

三、跨域请求处理

  1. 什么是跨域？

    跨域‌特指前端页面（如浏览器）与后端服务（SpringMVC 应用）因‌协议、域名或端口不一致‌触发同源策略限制，导致请求被浏览器拦截的现象：

    • 前端运行在 http://localhost:8080，后端接口部署在 http://api.example.com；
    • 前端通过 AJAX 调用后端接口时，浏览器因域名不同拦截请求并报错：CORS policy blocked。

    表现：浏览器控制台报错：Access to ... has been blocked by CORS policy，即使服务器返回响应，结果仍被拦截‌

四、异步处理

  1. 什么是异步处理？

    在 ‌SpringMVC‌ 中，‌异步处理‌ 是一种通过释放请求线程（Tomcat/Jetty 等容器的线程）以提升服务器吞吐量的机制。其核心思想是：

    • ‌同步阻塞模型‌：传统模式下，每个请求占用一个线程直到处理完成，线程资源被阻塞，无法处理新请求。
    • ‌异步非阻塞模型‌：将耗时操作（如 I/O 等待、外部服务调用）交给后台线程处理，释放容器线程，任务完成后通过回调或事件通知恢复响应。

  2. 处理方式：

    • ‌web.xml‌：增加<async-supported>

    • ‌方式一：返回Callable‌：耗时操作在后台线程执行，完成后返回结果。

      • ‌流程
        1. 主线程接收请求并返回 Callable。
        2. Spring 将 Callable 提交到 TaskExecutor 执行。
        3. 主线程释放，处理其他请求。
        4. 任务完成后，重新分配线程返回结果。

    • 方式二：返回DeferredResult：需要跨线程（如消息队列监听、外部事件触发）完成任务的场景。

      • ‌流程
        1. 主线程创建 DeferredResult 对象并返回。
        2. 主线程释放，处理其他请求。
        3. 外部线程（如消息监听器）调用 setResult() 设置结果。
        4. Spring 恢复响应，返回结果给客户端。

  3. 异步处理的优势：

    • ‌提升吞吐量‌：释放容器线程，避免线程池耗尽。
    • ‌优化资源利用率‌：适合处理 I/O 密集型任务（如网络请求、文件读写）。
    • ‌支持长任务处理‌：如实时数据推送、长时间计算任务。

  4. 适用场景：

    • ‌高并发请求‌：如电商秒杀场景，避免线程阻塞导致请求堆积。
    • ‌外部服务调用‌：如调用第三方 API 或微服务，等待响应期间释放资源。
    • ‌实时数据推送‌：如股票行情、聊天室消息的实时更新。
