一、Spring Cloud Alibaba

缘由：

Spring Cloud 的第一代实现 Spring Cloud Netflix 系列大部分组件都进入维护模式后，Spring Cloud 的第二代实现 Spring Cloud Alibaba 成为主流的微服务框架。

定位：

Spring Cloud Alibaba 是阿里巴巴结合自身丰富的微服务实践而推出的微服务开发的一站式解决方案，

组件：

  • Nacos：服务注册与发现（Alibaba）
  • Dubbo：服务调用与负载均衡
  • Sentinel：流量控制,熔断降级,系统负载保护（Alibaba）
  • Gateway：和Spring Cloud Netflix 使用的一样
  • Seata：分布式事务解决方案（Alibaba）
  • RocketMQ：高性能、高吞吐量的分布式消息和流计算平台

应用场景：

  • 大型复杂的系统，例如大型电商系统。
  • 高并发系统，例如大型门户网站、商品秒杀系统。
  • 需求不明确，且变更很快的系统，例如创业公司业务系统。

二、Spring Cloud 和 Spring Cloud Alibaba

  1. 服务注册与发现
    组件‌ ‌Spring Cloud‌ ‌Spring Cloud Alibaba‌
    ‌注册中心‌ Eureka（AP 模式）‌ ‌Nacos‌（AP/CP 模式可切换）‌
    ‌核心差异‌ pom引入即可，仅支持服务注册与发现‌ 需要单独下载，集成配置管理功能，支持动态配置刷新‌

  2. 服务调用与负载均衡
    ‌组件‌ ‌Spring Cloud‌ ‌Spring Cloud Alibaba‌
    ‌客户端负载‌ Ribbon + OpenFeign‌ ‌Dubbo‌（高性能 RPC）‌
    ‌协议支持‌ HTTP/REST‌ 支持 Dubbo 协议（TCP 长连接）‌

  3. 流量控制与熔断
    ‌组件‌ ‌Spring Cloud‌ ‌Spring Cloud Alibaba‌
    ‌熔断降级‌ Hystrix（已停更）‌ ‌Sentinel‌（实时监控、规则持久化）
    ‌核心特性‌ 简单熔断策略‌ 精细化流量控制（QPS/线程数/热点参数）‌

  4. 配置管理

    组件‌ ‌Spring Cloud‌ ‌Spring Cloud Alibaba‌
    ‌配置中心‌ Spring Cloud Config‌ ‌Nacos Config‌（与注册中心一体化）‌
    ‌优势‌ 需独立部署，功能基础‌ 动态配置推送、多环境隔离、版本回滚‌

  5. 分布式事务

    ‌组件‌ ‌Spring Cloud‌ ‌Spring Cloud Alibaba‌
    ‌解决方案‌ 无原生支持‌ ‌Seata‌（AT/TCC 模式）‌
    ‌适用场景‌ 需集成第三方框架‌ 跨服务事务一致性保障（如订单扣库存）

  6. 消息队列

    组件‌ ‌Spring Cloud‌ ‌Spring Cloud Alibaba
    ‌默认集成‌ RabbitMQ/Kafka‌ ‌RocketMQ‌（事务消息、顺序消息）‌‌
    ‌优势‌ 通用性强‌ 高吞吐、低延迟，适合金融级场景‌

  7. 生态系统与适用场景

    • Spring Cloud
      • 基于 Netflix OSS，适合公有云/私有云部署‌。
      • 社区成熟，文档丰富，但部分组件已停更（如 Hystrix）‌。
    • ‌Spring Cloud Alibaba
      • 深度整合阿里云生态，支持国内企业级需求（如高并发、分布式事务）‌。
      • 组件性能优化（如 Nacos、Sentinel）更适合本土化生产环境‌

  8. ‌总结选型建议‌

    • ‌选择 Spring Cloud‌：若项目依赖 Netflix 生态或需快速集成国际化云服务‌。
    • ‌选择 Spring Cloud Alibaba：
      • 需要高性能 RPC（Dubbo）、动态配置管理（Nacos）或分布式事务（Seata）‌。
      • 国内云环境部署，尤其是阿里云用户‌。

三、服务注册与发现-Nacos

  1. 组成：注册中心和配置中心Config的组合体

    组成部分 全称 描述
    Na naming/nameServer 即服务注册中心，与 Spring Cloud Eureka 的功能类似。
    co configuration 即配置中心，与 Spring Cloud Config+Spring Cloud Bus 的功能类似。
    s service 即服务，表示 Nacos 实现的服务注册中心和配置中心都是以服务为核心的。

    用来代替Eureka作为服务注册中心，代替 Spring Cloud Config 作为配置中心（动态刷新）。

  2. 特性：

    • 服务发现
      • 服务提供者向 Nacos 注册服务后，服务消费者可以在 Nacos 上通过 HTTP&API查找、发现服务。
    • 服务健康监测
      • 提供了一个健康检查仪表盘，能够帮助我们根据健康状态管理服务的可用性及流量
    • 动态配置服务
      • 管理所有环境的应用配置和服务配置
      • 消除了配置变更时重新部署应用和服务的需要
      • 服务按需弹性扩展变得更容易
      • Nacos 提供了一个简洁易用的 UI 帮助我们管理所有服务和应用的配置
        • 配置版本跟踪
        • 金丝雀发布
        • 一键回滚配置
        • 客户端配置更新状态跟踪
    • 动态DNS服务
      • 更容易地实现负载均衡、流量控制以及数据中心内网的简单 DNS 解析服务。
      • 管理服务的关联域名和可用的 IP:PORT 列表。
    • 服务及其元数据管理
      • 管理服务的描述、生命周期
      • 管理服务的静态依赖分析
      • 管理服务的健康状态
      • 管理服务的流量管理、路由及安全策略
      • 管理服务的 SLA 以及 metrics 统计数据

  3. 组件：client/server-客户端/服务端

    • Nacos Server

      • 可以作为服务注册中心，帮助Client 实现服务的注册与发现
      • 可以作为配置中心，帮助Client 在不重启的情况下，实现配置的动态刷新

    • Nacos Client

      • Client 通过添加依赖 spring-cloud-starter-alibaba-nacos-discovery，在服务注册中心（Nacos Server）中实现服务的注册与发现。
      • Client 通过添加依赖 spring-cloud-starter-alibaba-nacos-config，在配置中心（Nacos Server）中实现配置的动态刷新。

      • 服务注册中心（Register Service）：它是一个 Nacos Server，可以为服务提供者和服务消费者提供服务注册和发现功能。
      • 服务提供者（Provider Service）：它是一个 Nacos Client，用于对外服务。它将自己提供的服务注册到服务注册中心，以供服务消费者发现和调用。
      • 服务消费者（Consumer Service）：它是一个 Nacos Client，用于消费服务。它可以从服务注册中心获取服务列表，调用所需的服务。

    • Nacos 实现服务注册与发现的流程：

      • 下载 Nacos Server 并运行

        https://nacos.io/download/nacos-server/

      • 服务提供者 Nacos Client 启动时，会把服务以服务名（spring.application.name）的方式注册到服务注册中心（Nacos Server）

      • 服务消费者 Nacos Client 启动时，也会将自己的服务注册到服务注册中心

      • 服务消费者在注册服务的同时，它还会从服务注册中心获取一份服务注册列表信息，该列表中包含了所有注册到服务注册中心上的服务的信息（包括服务提供者和自身的信息）

      • 在获取了服务提供者的信息后，服务消费者通过 HTTP 或消息中间件远程调用服务提供者提供的服务

六、分布式事务-‌Seata

指南：https://c.biancheng.net/springcloud/seata.html

在分布式微服务架构中，几乎所有业务操作都需要多个服务协作才能完成。必须保证所有这些操作要么全部成功，要么全部失败。（如网购失败，钱也必须退回）

  • 分布式事务相关概念：
    • 事务：由一组操作构成的可靠、独立的工作单元，事务具备 ACID 的特性，即原子性、一致性、隔离性和持久性。
    • 本地事务：数据库管理系统 DBMS，例如 MySQL、Oracle 等，严格地支持 ACID 特性，高效可靠。本地事务只能对自己数据库的操作进行控制，对于其他数据库的操作则无能为力。
    • 全局事务：全局事务指的是一次性操作多个资源管理器完成的事务，由一组分支事务组成。
    • 分支事务：在分布式事务中，就是一个个受全局事务管辖和协调的本地事务。
  • 分布式事务是一个包含了若干个分支事务的全局事务，要么一起成功提交，要么一起失败回滚。
  • 核心组件：
    • **TC（Transaction Coordinator）**‌：事务协调器，全局事务的调度中心，负责维护全局事务状态和分支事务的提交/回滚‌。
    • ‌**TM（Transaction Manager）**‌：控制全局事务边界，通过 @GlobalTransactional 注解开启或结束全局事务‌。
    • ‌**RM（Resource Manager）**‌：管理分支事务资源（如数据库连接），向 TC 注册分支事务并上报状态‌。
  • Seata 的整体工作流程如下：
    1. TM 向 TC 申请开启一个全局事务，全局事务创建成功后，TC 会针对这个全局事务生成一个全局唯一的 XID；
    2. XID 通过服务的调用链传递到其他服务;
    3. RM 向 TC 注册一个分支事务，并将其纳入 XID 对应全局事务的管辖；
    4. TM 根据 TC 收集的各个分支事务的执行结果，向 TC 发起全局事务提交或回滚决议；
    5. TC 调度 XID 下管辖的所有分支事务完成提交或回滚操作

Seata 提供了 AT（最方便）、TCC、SAGA 和 XA 四种事务模式，可以快速有效地对分布式事务进行控制。

  • Seata AT 模式工作流程：
    • 一阶段（业务执行）
      • 拦截业务 SQL，生成 ‌**Before Image（前置镜像）**‌ 用于回滚‌
      • 提交本地事务，生成 ‌**After Image（后置镜像）**‌‌
    • 二阶段（全局提交/回滚）
      • 提交成功：删除镜像快照，释放锁‌
      • 提交失败：根据 Before Image 生成反向 SQL 回滚数据‌
