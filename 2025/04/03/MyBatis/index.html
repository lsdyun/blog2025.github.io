一、MyBatis

  1. MyBatis是一个开源、轻量级的数据持久化框架，内部封装了JDBC。

  2. MyBatis核心对象：

    （1）核心接口和类：

    • SqlSessionFactoryBuilder：临时工具类，用完即销毁，无状态，无需长期持有。

    • SqlSessionFactory：线程安全，与应用同生命周期，通常只初始化一次，避免重复解析配置；缓存数据库配置、Mapper 映射等元信息；由 IoC 容器管理其生命周期。

    • SqlSession：线程不安全，需在方法内创建并使用后立即关闭；代表一次数据库会话，用于执行 SQL、管理事务、获取 Mapper 接口。

      • 构建阶段‌：SqlSessionFactoryBuilder 读取配置文件 → 创建 SqlSessionFactory。

      • 运行阶段‌：SqlSessionFactory 生产 SqlSession → SqlSession 执行 SQL。

      • 销毁阶段‌：SqlSession 执行完毕后关闭，SqlSessionFactory 随应用关闭销毁。

    （2）MyBatis核心配置文件（mybatis-config.xml）

    • <properties resource="相对路径"/>:指定外部文件。

      或：

    • <environment>：环境配置，包括<transactionmanager>，<dataSource>。

    • <settings>：配置，例;

    • <typeAliases>：指定别名

    • <transactionManager>：事务管理器，提交、回滚。

    • <dataSource>：配置数据库中的连接属性，包括驱动程序名称、URL、用户名、密码。

    • <mapper>:指定SQL映射文件的路径。

    （3）SQL映射文件（mapper.xml）

    • 映射器-XML文件方式

      • 定义参数类型

      • 配置缓存

      • 提供 SQL 语句和动态 SQL

      • 定义查询结果和 POJO 的映射关系

      • 例：StudentMapper.xml

        还需要在Mybatis配置xml中添加资源路径，引入XML文件，生成映射器。

    • 映射器-注解方式

      • @Select

        注解 说明
        @Insert 实现新增
        @Delete 实现删除
        @Update 实现更新
        @Select 实现查询
        @Result 实现结果集封装
        @Results 可以与@Result 一起使用，封装多个结果集
        @ResultMap 实现引用@Results 定义的封装
        @One 实现一对一结果集封装，（@Result（one=@One）（）））
        @Many 实现一对多结果集封装，（@Result（many=@many）（）））

        更多注解

  3. 执行SQL方式

    • 通过 SqlSession 发送 SQL
    • 通过 SqlSession 获取 Mapper 接口，通过 Mapper 接口发送 SQL**（主流）**

  4. 标签

    • <select>

      • id：尽量和Mapper.java的方法名相同。

      • resultType：返回结果类型。（字段名与对象属性名完全一致时，自动映射到 entity）

      • resultMap：映射规则、级联。（字段名与属性名不一致，手动映射），一对一、一对多；返回的可以是resultMap也可以是是resultType

        MyBatis 的每一个查询映射的返回类型都是 resultMap,当提供的是resultType时，MyBatis会自动把值赋给resultType所指定的属性；当提供的是resultMap时，MyBatis会自动映射。

        resultMap 和 resultType 不能同时使用。

      • resultSetType：FORWARD_ONLY（只允许向前访问）、SCROLL_SENSITIVE（双向滚动，但不及时更新）、SCROLLJNSENSITIVE（双向滚动，及时更新）。

      • parameterType：int、float、String、JavaBean、Map。

      • fetchSize：限制获取的总记录数。

      • timeout：设置超时参数，单位秒，超时则报错。

  5. 动态SQL

    元素 作用 备注
    if 判断语句 单条件分支判断
    choose（when、otherwise） 相当于 Java 中的 switch case 语句 多条件分支判断
    trim、where 辅助元素 用于处理一些SQL拼装问题
    foreach 循环语句 在in语句等列举条件常用
    bind 辅助元素 拼接参数

    • <if>/<where>

      where的引入是为了解决1=1这个问题，当条件多个时，不采用1=1，会使SQL语句中出现多余的词如：AND。

      <where>自动去除条件中的AND/OR

    • <choose>/<when>/<otherwise>/<where>

    • <foreach>批量

      • item：表示集合中每一个元素进行迭代时的别名。
      • index：指定一个名字，表示在迭代过程中每次迭代到的位置。
      • open：表示该语句以什么开始（既然是 in 条件语句，所以必然以(开始）。
      • separator：表示在每次进行迭代之间以什么符号作为分隔符（既然是 in 条件语句，所以必然以,作为分隔符）。
      • close：表示该语句以什么结束（既然是 in 条件语句，所以必然以“)”开始）
      • collection：必选属性，分为3种情况list、array、map

    • <where>

      自动去除首条条件的 AND/OR，若无条件则忽略 WHERE。

    • <set>

      自动去除末尾逗号，用于更新语句。

    • <trim>自定义截取规则，代替<if>+<where>/<set>多条件查询

      属性 描述
      prefix 给SQL语句拼接的前缀，为 trim 包含的内容加上前缀
      suffix 给SQL语句拼接的后缀，为 trim 包含的内容加上后缀
      prefixOverrides 去除 SQL 语句前面的关键字或字符，该关键字或者字符由 prefixOverrides 属性指定。
      suffixOverrides 去除 SQL 语句后面的关键字或者字符，该关键字或者字符由 suffixOverrides 属性指定。

    • <bind>在 SQL 中复用表达式或处理复杂逻辑（如模糊查询），或者说拼接||。

      • value：对应传入实体类的某个字段，可以进行字符串拼接等特殊处理。
      • name：给对应参数取的别名。

  7. 其他

    • 参数处理与占位符

      **#{} vs ${}**‌

      • #{} 使用预编译防止 SQL 注入，适用于参数值传递（如 WHERE id = #{id}）‌。
      • ${} 直接替换 SQL 片段，适用于动态表名或排序字段（如 ORDER BY ${sortColumn}），需手动过滤风险参数‌。

    • 参数类型匹配‌

      • 简单类型（如 int, String）可直接传递；多参数需用 @Param 注解或封装为 Map/ENTITY。
      • 批量操作时，集合参数通过 <foreach> 遍历处理（如 IN 查询）‌。

    • ‌避免过度嵌套

      • 优先使用 <where>、<set> 自动处理前缀，减少冗余条件（如 AND/OR 前缀）‌。
      • 复杂逻辑用 <choose> 替代多层 <if>，提高可读性‌。

    • 性能优化

      • 减少动态标签数量，避免 SQL 语句频繁变化导致数据库硬解析‌。

      • 模糊查询用 <bind> 或数据库函数（如 CONCAT）统一处理，而非拼接 %‌。

        WHERE NAME LIKE CONCAT ('%',#{name},'%')

      • 分页时优先物理分页（如 LIMIT），而非逻辑分页（RowBounds）‌。

    • 字段名映射

      • mybatis-config.xml启用驼峰命名自动转换（mapUnderscoreToCamelCase=true），避免手动别名‌。

      • 复杂对象使用 <resultMap> 显式映射，尤其多表关联查询‌。

  3. 差别

    对比维度‌ ‌SpringMVC+MyBatis‌ ‌SpringBoot+MyBatis‌
    核心定位 传统企业级应用，需精细控制‌ 快速构建现代化应用，强调开箱即用‌
    配置复杂度 高（需手动配置多个组件）‌ 低（自动配置）‌
    依赖管理 手动维护，易冲突‌ Starter依赖自动管理‌
    部署方式 WAR包部署至外部容器‌ 内嵌容器，JAR包独立运行‌
    最佳适用场景 复杂业务定制、遗留系统维护‌ 微服务、快速迭代项目‌
    配置文件 多个XML文件（spring-mvc.xml等） 单一application.properties或YAML
    启动类 无，依赖外部容器启动 需@SpringBootApplication主类
    内嵌服务器 无，需外部Tomcat 支持内嵌Tomcat/Jetty
    依赖管理 手动管理版本 通过starter自动管理
    启动入口 依赖外部Servlet容器（如Tomcat）通过 web.xml 触发‌ 通过 @SpringBootApplication 主类调用 SpringApplication.run()‌
